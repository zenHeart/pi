# AGENTS.md

## 文档源码链接规范

- 所有涉及源码层面的引用必须使用可点击 Markdown link，不允许把 Markdown link 包在反引号中。
- 源码引用必须细化到具体行号，优先写成 `[file.ts#L123](/source-code/path/file.ts#L123)`。
- 如果正文需要展示函数名，函数名可以用反引号；对应源码位置必须另给可点击链接，例如：`foo()` 见 `[bar.ts#L10](/source-code/src/bar.ts#L10)`。
- 不要写成 `` `[file.ts#L123](/source-code/...)` ``、`` `[file.ts]` ``、`at [file.ts#L123](...)` 这类容易在 EPUB 中退化为文本的格式。
- EPUB 构建依赖 `build/preprocess.js` 将 `](/source-code/...#Lx)` 自动转换为 `](https://github.com/zenHeart/claude-code-sourcemap/blob/main/source-code/...#Lx)`；因此正文应保持 `/source-code/...#Lx` 形式，不能手写不完整的本地路径或缺行号路径。
- 修改分析文档后，应至少检查当前文件中是否存在反引号包裹的源码链接：`rg -n --fixed-strings '\`[' analysis/<file>.md`。

## 文档标题编号规范

- `analysis/[0-9][0-9]-*.md` 这类章节文件中进入 EPUB 目录的标题必须显式编号，避免目录里同级标题有的带序号、有的不带序号；`analysis/README.md` 不参与 EPUB 章节流，可保持导航页写法。
- H1 使用章节号，例如 `# 13. MCP 协议集成`。
- H2 使用一级章节号，例如 `## 5. 连接生命周期`。
- H3 必须跟随父 H2 编号，例如 `### 5.1 MCPConnectionManager`、`### 5.2 环境变量展开`。
- 同一文档内不得混用编号标题和裸标题；新增或移动小节时必须同步调整后续 H3 编号。
- 修改章节文档后，应检查当前文件是否还有裸 H3：`rg -n '^### (?![0-9]+\\.[0-9]+ )' analysis/<file>.md --pcre2`。

## 分析章节写作范式

- 源码分析章节的目标不是罗列代码，而是还原设计决策：先讲问题和约束，再讲机制、流程、源码证据、权衡和可迁移经验。
- 每个深入章节优先按“问题驱动 → 机制拆解 → 核心逻辑 → 源码落点 → 设计迁移 → 边界校准”的顺序组织。
- 章节开头应说明该模块解决什么系统问题：如果没有这个模块，系统会遇到什么不可接受的成本、风险或能力缺口。
- “如何解决”应写成机制组合，而不是只写函数调用链；优先说明协议/抽象、运行时、产品策略、安全边界如何协作。
- “核心逻辑”应提炼为设计不变量：无论实现细节如何变化，都不能破坏的责任边界、状态约束、权限约束、数据流约束。
- 源码映射不只标文件和函数，还要说明该源码承担的系统责任、输入、输出，以及在链路中的裁决边界。
- 设计经验应从 Claude Code 的具体实现抽象到通用工程原则，明确哪些经验可以迁移到自研 Agent，哪些只是 Claude Code 的产品选择。
- 每个复杂章节应尽量包含常见误解、失败模式或边界条件，避免读者学到过度简化甚至错误的模型。
- 复刻类章节应区分最小可用实现、生产级实现和可暂缓的高级能力，帮助读者按风险和收益分阶段落地。
- 复杂能力的时序图应显式标出阶段边界，例如“启动/发现”“用户触发/模型决策”“执行/结果回灌”；不要只画连续调用链，否则读者难以区分初始化、发现、触发、执行和后续上下文生命周期。
- 复杂能力章节优先回答 8 个生命周期问题：它解决什么系统问题；用户视角完整流程是什么；启动时加载什么、运行时再加载什么；模型能看到什么、Harness 私下保留什么；触发条件是用户主动、模型自动、生命周期事件还是外部调用；执行权在哪里；结果如何回灌；失败和安全边界是什么。
- 对照官方 Claude Code / Claude Platform 文档时，必须区分 Messages API、Claude Code CLI、Agent SDK、Managed Agents 四个抽象层。源码事实只说明本仓 `source-code/src` 中的 Claude Code 实现；SDK/API/托管形态的行为应标注为官方文档对照或设计迁移。
